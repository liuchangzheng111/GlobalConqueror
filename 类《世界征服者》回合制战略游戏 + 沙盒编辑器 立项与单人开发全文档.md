# 类《世界征服者》回合制战略游戏 + 沙盒编辑器 立项与单人开发全文档

**项目暂定名**：《全球征服者：沙盘版》

**开发引擎**：Unity

**开发规模**：单人独立开发

**文档版本**：V1.0

**核心定位**：轻量化回合制全球战略征服游戏，配套高自由度沙盒编辑器，兼顾经典征服玩法与玩家自定义创作需求

------

## 第一部分 立项核心准备工作

本部分为项目启动前必须完成的前置准备，核心目标是**锁定需求边界、规避单人开发风险、搭建可落地的技术与资源框架**，杜绝中途需求膨胀、技术卡点、项目烂尾。

### 第一章 项目定位与需求边界划定（单人开发核心前提）

#### 1.1 核心对标与差异化定位

- **对标产品**：《世界征服者》系列核心玩法（回合制大地图战略、国家运营、兵种作战、历史剧本征服）
- 核心差异化优势（单人破局点）：
  1. 深度原生沙盒编辑器，支持玩家全维度自定义地图、剧本、国家 / 兵种属性、胜利规则，解决原版编辑器自由度不足的痛点
  2. 轻量化单机玩法，无强制联机、无付费内购，适配 PC + 移动端双平台，降低玩家门槛
  3. 开源式数据结构，支持玩家自制剧本一键导入 / 导出，天然适配玩家社区传播
- **绝对不做的高成本方向（单人开发红线）**：3A 画质、多人联机对战、复杂外交 / 贸易 / 政治系统、全历史全国家剧本、内置社区与服务器、跨平台联机功能

#### 1.2 核心玩法闭环定义

锁定最核心的玩法循环，所有非核心功能全部延后至后续迭代：

```
回合开始 → 国家运营（资源生产、科技研发、部队建造）→ 部队部署/移动 → 战斗结算 → 回合事件触发 → 回合结束 → 胜利/失败判定
```

- **核心系统清单**：大地图网格系统、回合制管理系统、国家资源系统、兵种与部队系统、战斗结算系统、存档读档系统、基础 UI 系统
- **配套沙盒编辑器核心能力**：地图编辑、剧本编辑、基础属性编辑、剧本导入 / 导出功能

#### 1.3 MVP（最小可行产品）需求边界

| 分类 | MVP 必须完成的核心内容                                       | 绝对不纳入 MVP 的内容（放入后续迭代池）                      |
| :--- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 地图 | 六边形网格世界地图、基础地形（平原 / 山地 / 海洋 / 城市）、地块属性配置 | 高精度卫星地图、动态天气、地形破坏、多图层地图               |
| 玩法 | 单完整剧本（如 1v1 征服剧本）、回合制核心循环、基础资源 / 科技系统、3 类基础兵种（陆 / 海 / 空） | 多历史剧本、超 20 个可玩国家、复杂外交 / 结盟 / 宣战系统、将领系统、事件树系统 |
| 战斗 | 数值化自动战斗结算、基础攻防克制规则、占领机制               | 手动战术战斗、复杂兵种动画、战场特效、士气系统               |
| 配套 | 基础存档 / 读档、基础设置功能                                | 沙盒编辑器、成就系统、创意工坊、多语言本地化                 |
| 美术 | 简约 2D 图标化风格、基础 UI 框架、地块 / 兵种基础贴图        | 高精度立绘、复杂动画、全音效配乐、CG 剧情                    |

#### 1.4 目标平台与受众

- **首发平台**：PC（Steam/Itch.io），优先适配 Windows，MVP 完成后移植移动端（Android/iOS）
- **核心受众**：回合制战略游戏爱好者、历史剧本爱好者、沙盒创作型玩家、轻度策略游戏玩家

### 第二章 前置技术与资源准备

#### 2.1 技术栈选型（单人开发优先选稳定、低门槛、成熟方案，拒绝重复造轮子）

| 模块     | 选型方案                                        | 选型理由                                                     |
| :------- | :---------------------------------------------- | :----------------------------------------------------------- |
| 引擎版本 | Unity 2022.3 LTS 长期支持版                     | 稳定、BUG 少、文档 / 教程资源丰富、插件兼容性强，个人版完全免费（年收入超 10 万美元才需付费） |
| 代码架构 | 轻量化 MVC 架构                                 | 逻辑分层清晰、易维护、低学习成本，完全适配单人开发，不推荐复杂的 ECS 架构（开发成本过高） |
| 地图系统 | Unity 原生 Tilemap + 六边形网格规则             | 原生支持、开发效率高、无需额外付费插件，配套 Tiled 地图编辑器做复杂地图编辑 |
| 寻路系统 | A* Pathfinding Project 免费版                   | 成熟稳定、开箱即用，无需手动编写 A * 算法，节省大量开发时间  |
| 数据系统 | ScriptableObject（配置数据） + JSON（存档数据） | Unity 原生支持，无需搭建数据库，轻量化、易修改，完美适配编辑器自定义需求 |
| UI 系统  | Unity 原生 UGUI                                 | 教程资源多、兼容性强、开发门槛低，不推荐 UIToolkit（学习成本高，适配坑多） |
| 动画系统 | DoTween 免费版                                  | 轻量化、易上手，满足 UI、部队移动、战斗特效的基础动画需求，无需复杂的 Animator 状态机 |
| 版本控制 | Git + Gitee/GitHub                              | 免费、稳定，必备代码备份与版本回滚工具，杜绝代码丢失风险     |

#### 2.2 开发环境搭建

1. 安装 Unity Hub + Unity 2022.3 LTS，勾选 Windows Build Support、Android Build Support 模块
2. 配置 Git 环境，创建私有代码仓库，设置提交规范（如：功能提交 / BUG 修复 / 优化提交的命名规则）
3. 安装开发工具：Visual Studio 2022（代码编写）、Tiled（地图编辑）、Figma（UI 原型设计）、格式工厂（素材处理）
4. 搭建项目框架：提前划分文件夹目录（Scenes、Scripts、Art、Resources、Prefabs、Audio、Data），规范代码命名空间，避免后期项目混乱

#### 2.3 美术资源规划与获取（单人开发最大降本点）

单人开发核心原则：**优先使用合规免费商用素材，杜绝手动绘制全量美术资源，优先保证玩法，美术够用即可**

1. **美术风格锁定**：简约 2D 扁平化 / 图标化风格，对标《世界征服者》视觉体系，极致降低美术制作门槛
2. 免费商用资源渠道：
   - Unity Asset Store 免费商用 2D 地图 / UI / 兵种图标套件
   - [Kenney.nl](https://Kenney.nl) 全系列免费商用素材（CC0 协议，无版权风险）
   - Itch.io 独立开发者免费商用战略游戏素材包
   - 开源矢量世界地图素材（可修改为网格地块）
3. **资源优先级**：先搞定核心玩法所需的最小素材集（地块贴图、兵种图标、基础 UI 控件），美术优化全部放到 MVP 完成后

#### 2.4 音效与音乐资源准备

1. 优先使用 CC0 协议免费商用音效 / 背景音乐，渠道：Freesound、YouTube Audio Library、Unity Asset Store 免费音效包
2. MVP 阶段仅需基础音效（点击、战斗、回合切换），无需全量背景音乐，降低开发成本
3. 严格核对所有素材的授权协议，保留授权凭证，杜绝侵权风险

#### 2.5 必备付费插件（可选，总预算控制在 1000 元内）

仅推荐 MVP 完成后按需购买，不影响核心开发：

- DoTween Pro（45 美元）：进阶动画功能
- A* Pathfinding Project Pro（100 美元）：复杂地形寻路进阶功能
- Odin Inspector（55 美元）：编辑器开发神器，大幅降低沙盒编辑器开发成本

### 第三章 法律合规与知识产权准备

#### 3.1 知识产权与素材授权规范

1. 建立素材授权台账，记录所有素材的来源、授权协议、商用权限，保留授权文件
2. 绝对禁止使用无授权的素材（包括网上的图片、音乐、图标、历史剧本文案），避免上线后侵权纠纷
3. 游戏名称、LOGO 提前查重，避免与已发行游戏重名，预留商标注册空间
4. 核心代码与玩法规则无版权限制，但美术、音乐、文本内容必须 100% 合规

#### 3.2 平台上线合规要求

### 第四章 单人开发项目管理与风险预案

#### 4.1 代码与项目管理规范

1. **代码规范**：制定统一的命名规则、注释规范，每完成一个功能模块就进行代码重构，避免后期 “屎山代码” 无法维护
2. **版本控制**：每天开发结束提交代码，每个功能节点创建分支，完成后合并到主分支，重大版本更新打标签，预留回滚空间
3. **任务管理**：使用 Notion/Trello 搭建任务看板，拆分任务到最小单元（单个任务 2 小时内可完成），设置待办、进行中、已完成、需求池 4 个板块
4. **需求冻结机制**：MVP 开发阶段，所有新想法、新功能全部放入需求池，绝对不中途修改核心需求、新增功能，避免项目无限延期

#### 4.2 单人时间与精力管理

1. **固定开发节奏**：每日固定 3-4 小时核心开发时间，优先保证核心功能开发，避免熬夜开发导致的效率低下
2. **里程碑拆分**：以周为单位制定开发计划，每周日复盘进度，调整下周计划，预留 10%-20% 的缓冲时间应对技术卡点
3. **动力维持**：每个里程碑完成后设置奖励机制，定期发布开发日志，和玩家互动，获得正向反馈，避免单人开发的孤独感与倦怠感
4. **容错机制**：接受 “不完美”，MVP 阶段优先保证 “能玩、闭环”，而非 “完美、全功能”，杜绝过度优化导致的进度停滞

#### 4.3 核心风险识别与应对预案

| 风险类型     | 风险描述                                                     | 应对方案                                                     |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 技术卡点     | 核心系统（如地图、寻路、战斗）开发遇到无法解决的技术问题，导致进度停滞 | 1. 优先找现成插件 / 开源方案，不重复造轮子；2. 提前预留缓冲时间，单个技术问题攻关不超过 3 天；3. 加入 Unity 开发者社区、策略游戏开发社群，快速求助；4. 简化功能，先实现基础效果，后续迭代优化 |
| 需求膨胀     | 开发过程中不断新增功能，导致项目范围无限扩大，最终烂尾       | 1. 严格执行需求冻结机制，MVP 阶段不新增任何核心需求；2. 所有新想法全部放入需求池，MVP 完成后再评估优先级；3. 每周复盘进度，一旦出现偏离核心目标的情况，立即叫停 |
| 精力枯竭     | 单人开发周期长，出现倦怠、动力不足，导致项目停滞             | 1. 拆分任务到最小单元，每天都有可交付的成果，获得正向反馈；2. 固定休息时间，每周至少休息 1 天，不强制加班；3. 定期发布开发日志，和玩家互动，获得外部动力；4. 遇到瓶颈时，优先做简单的任务，恢复开发节奏 |
| 版权风险     | 素材侵权、内容违规，导致平台下架、法律纠纷                   | 1. 所有素材必须核对商用授权，保留授权凭证；2. 不使用受版权保护的历史 IP、文本、音乐；3. 上线前完成合规自查，规避敏感内容 |
| 上线后无玩家 | 游戏完成后无人问津，开发成果白费                             | 1. 开发阶段就开始发布开发日志，积累核心玩家；2. 先发布免费 demo，收集反馈优化游戏，再发布完整版；3. 入驻策略游戏玩家社区，和玩家深度沟通，适配核心受众需求 |

## 第二部分 单人开发项目计划书

### 项目总览

- **总开发周期**：约 10 个月（42 周），分为 7 个核心阶段，预留 20% 缓冲时间
- **核心里程碑**：玩法原型验证 → MVP 闭环 → 编辑器完成 → 内容填充 → 测试优化 → 正式上线
- **核心优先级**：玩法闭环 > 编辑器功能 > 内容填充 > 美术 / 音效优化 > 附加功能

### 分阶段详细开发计划

#### 阶段 1：预研与技术验证阶段（2 周，第 1-2 周）

**核心目标**：验证核心技术可行性，完成项目立项全文档，冻结 MVP 需求

| 时间    | 核心任务                                                     | 交付物                                                       | 验收标准                                                   |
| :------ | :----------------------------------------------------------- | :----------------------------------------------------------- | :--------------------------------------------------------- |
| 第 1 周 | 1. 完成竞品分析与需求文档定稿；2. 搭建 Unity 项目框架与开发环境；3. 完成技术栈选型与插件调研；4. 六边形网格地图系统技术预研 | 1. 项目需求文档 V1.0；2. 初始化 Unity 项目框架；3. 技术选型方案文档 | 项目目录规范、开发环境可正常运行、地图技术方案验证可行     |
| 第 2 周 | 1. 完成 3 个核心技术点原型开发：网格地图移动、回合制循环、基础战斗结算；2. 完成美术 / 音效资源清单与获取渠道确认；3. 完成任务看板搭建，拆分 MVP 阶段全量任务 | 1. 可运行的技术验证原型；2. 资源清单与采购计划；3. 全周期任务拆解看板 | 原型可实现地块移动、回合切换、基础战斗数值结算，无致命 BUG |

#### 阶段 2：MVP 核心玩法闭环开发阶段（12 周，第 3-14 周）

**核心目标**：完成所有 MVP 核心系统开发，实现完整的单局游戏闭环，可正常通关一局游戏

| 时间        | 核心模块            | 核心任务                                                     |
| :---------- | :------------------ | :----------------------------------------------------------- |
| 第 3-4 周   | 地图系统            | 1. 完成六边形网格世界地图基础开发；2. 实现地形系统、地块属性（资源、归属、防御）；3. 实现地图缩放、拖拽、地块选中基础交互 |
| 第 5-6 周   | 回合制与国家系统    | 1. 完成回合制管理系统（回合切换、回合事件触发）；2. 实现国家资源系统（生产、消耗、存储）；3. 实现基础科技研发系统 |
| 第 7-8 周   | 兵种与部队系统      | 1. 完成陆 / 海 / 空 3 类基础兵种属性配置；2. 实现部队创建、部署、移动、合并 / 拆分功能；3. 接入 A * 寻路系统，实现部队合规路径移动 |

**阶段交付物**：可完整游玩的 MVP 游戏 demo，单剧本全流程无阻断，核心系统全部落地

**验收标准**：可正常完成一局 30-60 分钟的征服游戏，无致命 BUG，所有核心功能正常运行

| 时间        | 核心模块             | 核心任务                                                     |

### 项目验收与复盘机制
